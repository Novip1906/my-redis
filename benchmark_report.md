# Отчет о бенчмарке форматов AOF (Append Only File)

## Среда тестирования

| Параметр | Значение |
| :--- | :--- |
| CPU | Apple M1 Pro |
| RAM | 16 GB |
| OS | macOS 26.3.1 |
| Go | 1.25.1 darwin/arm64 |

## Методология

Бенчмарк выполняет **последовательную запись 1 000 000** команд из набора разнообразных строк (ASCII, кириллица, длинные строки, несколько аргументов) в файл для каждого из 4 форматов сериализации. Запись идёт синхронно, без конкурентности, чтобы измерить чистую производительность сериализации и записи в файл. По окончании записи вызывается `file.Sync()` для гарантии сброса буферов на диск.

## Анализ форматов

### 1. Text (Простой текст)
*   **Плюсы:** Максимальная компактность (всего +1 байт `\n` на команду) и читаемость. Наилучшие показатели по времени чтения (~140-160 мс), что объясняется аппаратной оптимизацией функции `ReadString` в пакете `bufio`. Время записи сопоставимо с бинарным форматом благодаря эффективности выделения памяти функцией `append` в Go.
*   **Минусы:** Отсутствие бинарной безопасности. Работа парсера нарушается при наличии символа переноса строки (`\n`). Очевидным решением для корректного хранения таких команд является экранирование, которое вносит вычислительные задержки и отрицательно сказывается на скорости выполнения операций.

### 2. Binary (Бинарный)
*   **Плюсы:** Гарантированная консистентность и поддержка произвольных бинарных данных без экранирования. Высокая скорость записи (~610-800 мс) на одном уровне с текстовым форматом, обусловленная отсутствием дополнительных операций сериализации.
*   **Минусы:** Отсутствие человекочитаемости. Наличие накладных расходов в 4 байта на каждую команду, что может быть значимо только при сохранении огромного количества мелких ключей. Скорость чтения уступает текстовому формату, так как для реконструкции полезной нагрузки требуются два последовательных вызова функции `io.ReadFull`.

### 3. JSON
*   **Плюсы:** Строгая структурированность, кроссплатформенность, высокая читаемость.
*   **Минусы:** Избыточность данных за счет ключей объекта (например, `"cmd":`). Наибольшие затраты процессорного времени при сериализации и десериализации (более 2 с) из-за использования пакета `encoding/json` и рефлексии (глубокой интерпретации типов).

### 4. RESP (Redis Serialization Protocol)
*   **Плюсы:** Протокол стандарта де-факто для Redis, поддерживающий бинарно-безопасные строки. Относительный размер служебных метаданных уменьшается с ростом размера сохраняемой команды.
*   **Минусы:** Наличие избыточных управляющих символов (`*`, `$`, `\r\n`). В условиях текущего бенчмарка демонстрирует более медленную скорость записи из-за того, что парсинг исходной строки на токены выполняется в теле самого бенчмарка. В промышленной эксплуатации парсинг происходит в сетевом цикле обмена (event loop), что значительно улучшает показатели производительности формат.

## Итоговая рекомендация
Для реализации собственного Redis-подобного хранилища **бинарный формат** является лучшим выбором по балансу скорости восстановления и надежности данных. **Text** формат стоит оставить только для отладки, так как он требует сложного экранирования для реальных данных. **RESP** следует использовать только если важна прямая совместимость с клиентами Redis.
