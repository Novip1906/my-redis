# Отчет о бенчмарке форматов AOF (Append Only File)

## Среда тестирования

| Параметр | Значение |
| :--- | :--- |
| CPU | Apple M1 Pro |
| RAM | 16 GB |
| OS | macOS 26.3.1 |
| Go | 1.25.1 darwin/arm64 |

## Методология

Бенчмарк выполняет **последовательную запись 1 000 000** команд из набора разнообразных строк (ASCII, кириллица, длинные строки, несколько аргументов) в файл для каждого из 4 форматов сериализации. Запись идёт синхронно, без конкурентности, чтобы измерить чистую производительность сериализации и записи в файл. По окончании записи вызывается `file.Sync()` для гарантии сброса буферов на диск.


## Результаты тестирования (1 000 000 записей)

| Формат | Запись (сек) | Чтение/AOF Restore (мс) | Размер файла (МБ) |
| :--- | :--- | :--- | :--- |
| **Binary** | **0.54** | 235.0 | 305.9 |
| **Text** | 0.59 | **150.5** | **303.0** |
| **Gzip Binary** | 0.77 | 193.0 | 1.43 |
| **JSON** | 1.17 | 2067.2 | 313.4 |
| **RESP** | 1.70 | 457.1 | 320.4 |

*Примечание: Чтение проводилось сразу после записи.*

### Анализ разницы в размере файлов (Binary vs Text)
Разница в объеме файлов Binary и Text форматов вызвана структурой метаданных:
*   **Text формат** использует **1 байт** разделителя (`\n`) на каждую команду.
*   **Binary формат** сохраняет длину команды (`uint32`), что добавляет **4 байта** метаданных на каждую запись.

При объеме в 1 000 000 записей это дает разницу в 3 000 000 байт (~2.86 МБ), что подтверждается тестами: 303.0 МБ (Text) + 2.86 МБ ≈ 305.9 МБ (Binary).

## Анализ форматов

### 1. Text (Простой текст)
*   **Плюсы:** Компактность (+1 байт на команду) и читаемость. Высокая скорость чтения (~150 мс) за счет эффективного сканирования разделителя в `bufio.Reader`.
*   **Минусы:** Отсутствие бинарной безопасности. Наличие символа `\n` в данных требует экранирования, что усложняет парсинг и вносит вычислительные задержки в реальных условиях.

### 2. Binary (Бинарный)
*   **Плюсы:** Бинарная безопасность без дополнительного кодирования. Стабильная производительность записи (~540 мс), сопоставимая с текстовым форматом при использовании буферизированного ввода-вывода.
*   **Минусы:** Нечитаемость данных без специальных инструментов. Оверхед на хранение длины каждой команды (4 байта).

### 3. JSON
*   **Плюсы:** Стандартизация и переносимость данных.
*   **Минусы:** Избыточность из-за хранения ключей объектов. Значительные затраты ресурсов на сериализацию и десериализацию (~2 с) в стандартной библиотеке Go.

### 4. RESP (Redis Serialization Protocol)
*   **Плюсы:** Совместимость с протоколом Redis и поддержка произвольных бинарных данных.
*   **Минусы:** Наличие управляющих символов увеличивает объем файла. В рамках данного бенчмарка показывает меньшую производительность из-за накладных расходов на парсинг и форматирование строк.

### 5. Gzip Binary (Потоковое сжатие)
*   **Плюсы:** Значительное сокращение занимаемого пространства на диске (с 305 МБ до 1.43 МБ) при работе с повторяющимися данными. Скорость восстановления сопоставима с текстовым форматом, так как уменьшение объема I/O нивелирует затраты CPU на распаковку.
*   **Минусы:** Дополнительная нагрузка на CPU при записи. Формат требует последовательного чтения; произвольный доступ к середине файла (Random Access) без перестроения индексов или использования точек сброса невозможен. Эффективность сжатия сильно зависит от политики сброса буферов на диск (`fsync`).

## Итоговая рекомендация
Для реализации Redis-подобного хранилища **бинарный формат** является наиболее сбалансированным решением с точки зрения надежности и простоты парсинга. Применение **сжатия (Gzip)** целесообразно использовать в задачах архивации или фоновой перезаписи AOF-файлов для минимизации дискового пространства. Текстовый формат рекомендуется ограничить отладочными целями.
