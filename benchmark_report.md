# Отчет о бенчмарке форматов AOF (Append Only File)

## Среда тестирования

| Параметр | Значение |
| :--- | :--- |
| CPU | Apple M1 Pro |
| RAM | 16 GB |
| OS | macOS 26.3.1 |
| Go | 1.25.1 darwin/arm64 |

## Методология

Бенчмарк выполняет **последовательную запись 1 000 000** команд из набора разнообразных строк (ASCII, кириллица, длинные строки, несколько аргументов) в файл для каждого из 4 форматов сериализации. Запись идёт синхронно, без конкурентности, чтобы измерить чистую производительность сериализации и записи в файл. По окончании записи вызывается `file.Sync()` для гарантии сброса буферов на диск.


## Результаты тестирования (1 000 000 записей)

| Формат | Запись (сек) | Чтение/AOF Restore (мс) | Размер файла (МБ) |
| :--- | :--- | :--- | :--- |

## Анализ форматов

### 1. Text (Простой текст)
*   **Плюсы:** Максимально компактный размер (всего +1 байт `\n` на команду). Идеальная читаемость. **Высокая скорость чтения** (на уровне или выше бинарного формата для простых строк).
*   **Минусы:** **Ненадежность.** Текущий парсер сломается, если значение содержит перенос строки. Запись медленнее бинарного из-за накладных расходов на форматирование строк.

### 2. Binary (Бинарный)
*   **Плюсы:** **Максимальная производительность при записи.** Минимальные накладные расходы на сериализацию. **Надежность:** корректно сохраняет любые бинарные данные без экранирования.
*   **Минусы:** Нечитаем вручную. На больших данных накладные расходы (4 байта) становятся пренебрежимо малыми, но всё же присутствуют.

### 3. JSON
*   **Плюсы:** Структурированность, поддержка любыми языками, читаемость.
*   **Минусы:** Избыточность (повторяющиеся ключи `"cmd":`). Самый высокий CPU overhead из-за работы `json.Marshal/Unmarshal` через рефлексию. 

### 4. RESP (Redis Serialization Protocol)
*   **Плюсы:** Поддерживает произвольные бинарные данные, структурирован как массив аргументов.
*   **Минусы:** Присутствует избыточность из-за `*`, `$`, `\r\n`. Однако на **больших** данных (десятки KB) эта избыточность становится незначительной (менее 5% в данном тесте).

## Итоговая рекомендация
Для реализации собственного Redis-подобного хранилища **бинарный формат** является лучшим выбором по балансу скорости восстановления и надежности данных. **Text** формат стоит оставить только для отладки, так как он требует сложного экранирования для реальных данных. **RESP** следует использовать только если важна прямая совместимость с клиентами Redis.
