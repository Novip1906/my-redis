# Отчет о бенчмарке форматов AOF (Append Only File)

## Среда тестирования

| Параметр | Значение |
| :--- | :--- |
| CPU | Apple M1 Pro |
| RAM | 16 GB |
| OS | macOS 26.3.1 |
| Go | 1.25.1 darwin/arm64 |

## Методология

Бенчмарк реализован с использованием стандартного пакета `testing` языка Go. Используется функция `testing.B` для точного замера времени (в наносекундах `ns/op`), размеров на диске (`disk_bytes/op`), а также показателей потребления оперативной памяти (`B/op`, `allocs/op`). Запись идёт синхронно, без конкурентности, для измерения чистой производительности сериализации и записи в файл (с обязательным вызовом `file.Sync()`). Чтение замеряет скорость восстановления данных алгоритмом и десериализацию.

Тестовые данные представлены набором коротких, средних и очень длинных строк, а также ASCII и кириллистических символов вперемешку.

## Анализ форматов

### 1. Binary (Бинарный)
*   **Преимущества:** Наименьшее время записи (434 ns/op) и чтения (210 ns/op) среди тестируемых форматов. Формат бинарно безопасен. Низкое потребление памяти при записи (4 байта на аллокацию).
*   **Недостатки:** Требует на 1% больше дискового пространства по сравнению с текстовым форматом из-за добавления 4-байтового префикса длины. Данные не поддаются прямому чтению пользователем.

### 2. Text (Простой текст)
*   **Преимущества:** Высокая плотность упаковки без использования сжатия (+1 байт на запись). Минимальная нагрузка на сборщик мусора (1 аллокация на операцию при чтении и записи).
*   **Недостатки:** Отсутствие бинарной безопасности. Наличие символа переноса строки в сохраняемых данных потенциально приводит к ошибкам парсинга.

### 3. Gzip Binary (Потоковое сжатие)
*   **Преимущества:** Крайне малый объем данных на диске для тестовой выборки с высокой степенью дубликатов (1.5 байта на команду). Время чтения сопоставимо с несжатым бинарным форматом за счет снижения I/O нагрузки.
*   **Недостатки:** Повышенная нагрузка на CPU при сжатии (на 77% медленнее записи несжатого формата). Ограничения в организации произвольного доступа к данным (Random Access).

### 4. JSON
*   **Преимущества:** Стандартизация, строгая структурированность и переносимость данных между системами.
*   **Недостатки:** Низкие показатели скорости десериализации (2040 ns/op). Высокое потребление оперативной памяти из-за использования пакета `encoding/json` (1664 байта и 8 аллокаций на операцию чтения).

### 5. RESP (Redis Serialization Protocol)
*   **Преимущества:** Совместимость с протоколом и экосистемой Redis. Поддержка сохранения сырых байтовых массивов.
*   **Недостатки:** Наихудшая скорость записи среди тестируемых алгоритмов (1624 ns/op). Существенные накладные расходы при аллокации памяти за счет обилия служебных символов и форматирования строк (1082 байта и 9 аллокаций на записываемую команду).

## Итоговый вывод
По результатам проведенного тестирования, **Бинарный (Binary)** формат является оптимальным решением для реализации локального AOF-хранилища: он обеспечивает максимальную производительность ввода-вывода и гарантию бинарной безопасности. В сценариях ротации или архивации AOF-журналов целесообразно применение сжатия **Gzip Binary** для экономии дискового пространства. Применение форматов JSON и RESP в основном цикле записи демонстрирует заметную деградацию производительности, что связано с высокими накладными расходами на парсинг и выделение памяти на куче (Heap).
