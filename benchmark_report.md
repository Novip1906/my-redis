# Отчет о бенчмарке форматов AOF (Append Only File)

В ходе тестирования была проведена оценка производительности 5 различных форматов сериализации данных для системы логирования команд (AOF). Тест выполнялся в конкурентной среде: **1 000 000 записей** команды `SET my_test_key value`.

## Результаты тестирования

| Формат | Время (сек) | Размер файла (байт) |
| :--- | :--- | :--- |
| **Binary** | **3.33s** | 25 000 000 |
| **Text** | 3.54s | **22 000 000** |
| **RESP** | 3.51s | 42 000 000 |
| **JSON** | 4.09s | 32 000 000 |

---

## Анализ форматов

### 1. Binary (Бинарный)
*   **Плюсы:** Самый быстрый результат. Минимальные накладные расходы на сериализацию (только длина строки + сами данные).
*   **Минусы:** Данные нечитаемы для человека без специальных инструментов.

### 2. Text (Простой текст)
*   **Плюсы:** Минимальный размер файла (только команда + `\n`). Легко отлаживать.
*   **Минусы:** Сложно парсить, если значения ключей содержат спецсимволы или пробелы (требуется экранирование).

### 3. RESP (Redis Serialization Protocol)
*   **Плюсы:** Стандарт для Redis. Позволяет использовать существующие инструменты парсинга и обеспечивать совместимость.
*   **Минусы:** Самый большой размер файла (из-за обилия служебных символов `\r\n`, `$`, `*`). Скорость чуть ниже бинарного из-за сложности формирования строки.

### 4. JSON
*   **Плюсы:** Универсальность, читаемость.
*   **Минусы:** Самый медленный (из-за ресурсозатратной рефлексии при `json.Marshal`). Большой размер из-за ключей объектов.

## Итоговая рекомендация
Для реализации лучшим выбором будет **RESP**, если важна совместимость с экосистемой Redis, либо **Binary**, если во главу угла ставится максимальная производительность и экономия места.
