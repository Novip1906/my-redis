# Отчет о бенчмарке форматов AOF (Append Only File)

## Среда тестирования

| Параметр | Значение |
| :--- | :--- |
| CPU | Apple M1 Pro |
| RAM | 16 GB |
| OS | macOS 26.3.1 |
| Go | 1.25.1 darwin/arm64 |

## Методология

Бенчмарк выполняет **последовательную запись 1 000 000** команд из набора разнообразных строк (ASCII, кириллица, длинные строки, несколько аргументов) в файл для каждого из 4 форматов сериализации. Запись идёт синхронно, без конкурентности, чтобы измерить чистую производительность сериализации и записи в файл. По окончании записи вызывается `file.Sync()` для гарантии сброса буферов на диск.


## Результаты тестирования (1 000 000 записей)

| Формат | Запись (сек) | Чтение/AOF Restore (мс) | Размер файла (МБ) |
| :--- | :--- | :--- | :--- |
| **Binary** | 0.55 | **165.5** | 305.9 |
| **Text** | **0.54** | 188.9 | **303.0** |
| **JSON** | 1.01 | 1957.4 | 313.4 |
| **RESP** | 1.79 | 449.3 | 320.4 |

*Примечание: Чтение проводилось сразу после записи.*

### Анализ разницы в размере файлов (Binary vs Text)
Несмотря на название «бинарный», данный формат в текущей реализации занимает больше места, чем простой текстовый. Это объясняется структурой метаданных:
*   **Text формат** добавляет только **1 байт** разделителя (`\n`) на каждую запись.
*   **Binary формат** сохраняет длину команды в виде `uint32`, что добавляет **4 байта** метаданных на каждую запись.

При объеме в 1 000 000 записей разница составляет ровно 3 000 000 байт (~2.86 МБ), что подтверждается результатами теста: 303.0 МБ (Text) + 2.86 МБ ≈ 305.9 МБ (Binary).

## Анализ форматов

### 1. Text (Простой текст)
*   **Плюсы:** Компактность (+1 байт `\n` на команду) и читаемость. Высокая скорость чтения (~180-190 мс) и общая пропускная способность записи на уровне бинарного формата, оптимизированная за счет буферизованного вывода (`io.Writer`).
*   **Минусы:** Формат не является бинарно-безопасным (наличие символа `\n` в полезной нагрузке нарушает структуру лога). Переход на безопасное хранение потребует экранирования данных, что приведет к увеличению накладных расходов CPU на чтение и запись.

### 2. Binary (Бинарный)
*   **Плюсы:** Гарантированная консистентность бинарных данных без дополнительного экранирования. Высокая скорость записи (~550 мс) на одном уровне с текстовым форматом. Чтение блоков фиксированного размера (`io.ReadFull`) в данном тесте оказалось эффективнее сканирования текстовых разделителей.
*   **Минусы:** Данные не поддаются прямому чтению человеком без специализированных утилит. Статический оверхед на метаданные составляет 4 байта на каждую команду.

### 3. JSON
*   **Плюсы:** Строгая структурированность, переносимость формата и высокая читаемость схемы.
*   **Минусы:** Избыточный расход дискового пространства за счет сохранения имен ключей. Относительно высокие затраты процессорного времени при операциях чтения и записи (~2 с) в связи с особенностями стандартного пакета `encoding/json`.

### 4. RESP (Redis Serialization Protocol)
*   **Плюсы:** Стандарт совместимости с экосистемой Redis. Формат бинарно-безопасен, при этом доля метаданных снижается пропорционально размеру команд.
*   **Минусы:** Присутствуют накладные расходы на управляющие протокольные символы (`*`, `$`, `\r\n`). В изолированном тесте показывает более низкую скорость из-за необходимости дополнительной разбивки текстовой строки на структурные элементы перед сериализацией. В промышленной эксплуатации парсинг происходит в сетевом цикле обмена (event loop), что значительно улучшает показатели производительности формат.

## Итоговая рекомендация
Для реализации собственного Redis-подобного хранилища **бинарный формат** является лучшим выбором по балансу скорости восстановления и надежности данных. **Text** формат стоит оставить только для отладки, так как он требует сложного экранирования для реальных данных. **RESP** следует использовать только если важна прямая совместимость с клиентами Redis.
